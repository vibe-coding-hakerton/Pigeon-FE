## 문서별 상세 설명

### 2.1 시스템 아키텍처 (`ARCHITECTURE.md`)
```
포함 내용:
- 전체 시스템 구성도 (다이어그램)
- 프론트엔드 컴포넌트 구조
- 백엔드 모듈 구조
- 디렉토리/파일 구조 (프로젝트 스켈레톤)
- 기술 스택 상세 및 버전
- 외부 서비스 연동 (LLM API)
```

### 2.2 API 명세서 (`API_SPEC.md`)
```
포함 내용:
- 모든 엔드포인트 목록
- 각 API별 상세:
  - HTTP Method, URL
  - Request Body/Query Params (타입, 필수 여부)
  - Response Body (성공/실패)
  - 에러 코드 정의
- 인증 방식 (MVP는 없음)
```

### 2.3 데이터베이스 설계 (`DATABASE.md`)
```
포함 내용:
- ERD (Entity Relationship Diagram)
- 테이블 정의 (컬럼, 타입, 제약조건)
- 관계 설명
- 인덱스 정의
- 초기 데이터 (시드)
```

### 2.4 UI/UX 설계 (`UI_DESIGN.md`)
```
포함 내용:
- 페이지별 상세 와이어프레임
- React 컴포넌트 트리
- 상태(State) 설계
- 사용자 인터랙션 흐름
- 반응형 고려사항
```

### 3.1 프롬프트 설계 (`PROMPTS.md`)
```
포함 내용:
- 시스템 프롬프트
- 분류 요청 프롬프트 템플릿
- 출력 포맷 (JSON 스키마)
- Few-shot 예시
- 에지 케이스 처리 방법
```

### 3.2 샘플 데이터 (`sample_mails.json`)
```
포함 내용:
- 다양한 카테고리의 샘플 메일 10~20개
  - 업무 메일 (회의, 보고서, 공지)
  - 개인 메일 (쇼핑, 예약)
  - 알림 메일 (GitHub, Jira)
  - 뉴스레터
  - 스팸/광고
- 각 메일의 기대 분류 결과 (테스트용)
```

### 4.1 작업 계획서 (`WORK_PLAN.md`) - **Claude Squad 병렬 작업용**
```
Claude Code가 작업 순서를 이해하고 Squad로 병렬 작업을 진행하기 위한 핵심 문서

포함 내용:
- 전체 개발 Phase 정의
- Phase별 작업 목록
- 작업 간 의존성 그래프
  - 어떤 작업이 선행되어야 하는지
  - 어떤 작업들이 병렬 가능한지
- Squad 역할 분담
  - Backend Agent: API, DB, LLM 연동
  - Frontend Agent: UI 컴포넌트, 상태 관리
  - Integration Agent: 연동, 테스트
- 각 작업의 예상 산출물
- 체크포인트 (동기화 시점)

예시 구조:
┌─────────────────────────────────────────────────────┐
│ Phase 1: 프로젝트 세팅                               │
│ ├── [BE] FastAPI 프로젝트 초기화      ─┐            │
│ ├── [FE] React 프로젝트 초기화        ─┼─ 병렬 가능  │
│ └── [BE] DB 스키마 생성               ─┘            │
│                    ↓ (Phase 1 완료 후)              │
│ Phase 2: 핵심 기능 개발                              │
│ ├── [BE] 메일 CRUD API      ──┐                    │
│ ├── [BE] 폴더 CRUD API      ──┼─ 병렬 가능         │
│ ├── [AI] LLM 분류 서비스    ──┘                    │
│ │            ↓ (API 완료 후)                        │
│ ├── [FE] 폴더 트리 UI       ──┐                    │
│ └── [FE] 메일 목록 UI       ──┴─ 병렬 가능         │
└─────────────────────────────────────────────────────┘
```

### 4.2 DoD/AEC 정의 (`DOD_AEC.md`) - **완료 판단용**
```
Claude가 자율적으로 완료를 판단하기 위한 핵심 문서

포함 내용:
- 전역 DoD (모든 Story 공통)
  - 코드가 린트 에러 없이 통과
  - 관련 테스트 통과
  - 기능이 정상 동작 확인

- Story별 AEC (인수 조건)
  예: "메일 목록 조회 API"
  - [ ] GET /api/mails 호출 시 200 응답
  - [ ] folderId 쿼리 파라미터로 필터링 가능
  - [ ] 응답에 mails 배열과 total 필드 포함
  - [ ] 빈 폴더일 경우 빈 배열 반환
```

### 4.3 테스트 시나리오 (`TEST_SCENARIOS.md`)
```
포함 내용:
- 단위 테스트 케이스 목록
- E2E 테스트 시나리오 (Playwright)
  - 시나리오명, 단계, 기대 결과
- 엣지 케이스 목록
```

### 5.1 Jira 이슈 목록 (`JIRA_ISSUES.md`)
```
포함 내용:
- Epic 목록 (ID, 제목, 설명)
- Story 목록 (Epic 하위, ID, 제목, 설명, DoD, AEC)
- Task 목록 (Story 하위, 구체적 작업 단위)
- 우선순위 및 의존성
```

### 5.2 기술 결정 기록 (`DECISIONS.md`)
```
포함 내용:
- 결정 사항 목록
  - 결정: FastAPI 사용
  - 대안: Node.js + Express
  - 이유: Python LLM 라이브러리 풍부, 빠른 개발
  - 날짜: 2025-12-09
```
